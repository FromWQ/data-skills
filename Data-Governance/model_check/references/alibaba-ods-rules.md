# 阿里巴巴ODS层规范详解

## ODS层定义

ODS（Operational Data Store）层是数据仓库的贴源层，位于业务系统和数据仓库之间，是数据进入数仓的第一站。

## ODS层职责

### 1. 数据同步

- 从各业务系统抽取数据
- 保持数据的原始形态
- 提供统一的访问接口

### 2. 历史归档

- 保存业务系统的历史数据
- 支持数据回溯和审计
- 降低源系统压力

### 3. 数据隔离

- 屏蔽源系统变化
- 统一数据格式
- 提供数据质量保障

## 表设计规范

### 命名规范

```
ods_{源系统}_{表名}_{增量标识}
```

| 组成部分 | 规范要求 | 示例 |
|---------|---------|-----|
| 前缀 | 固定为ods | ods_ |
| 源系统 | 2-20字符，小写字母+下划线 | crm, erp, order |
| 表名 | 与源表对应，小写+下划线 | user_info, order_detail |
| 增量标识 | di/df/hi | di, df, hi |

### 字段规范

**命名规则:**
- 使用小写字母
- 单词间用下划线分隔
- 长度不超过32字符
- 见名知意

**注释规范:**
- 所有字段必须有注释
- 注释内容完整准确
- 包含业务含义说明

**主键规范:**
- 明确标注主键字段
- 主键不可为空
- 主键不可重复

### 分区规范

| 同步策略 | 分区字段 | 说明 |
|---------|---------|-----|
| 日增量 | dt | 按日分区 |
| 日全量 | dt | 按日分区，每日全量快照 |
| 小时增量 | dt, hh | 按小时分区 |

分区格式要求:
- dt格式: yyyyMMdd (如 20240101)
- hh格式: HH (如 08)

## 数据同步规范

### 同步策略选择

| 策略 | 适用场景 | 优势 | 劣势 |
|-----|---------|-----|-----|
| 日增量 | 有增量字段，数据量大 | 存储效率高 | 需增量标识 |
| 日全量 | 数据量小，无增量标识 | 实现简单 | 存储开销大 |
| 小时增量 | 实时性要求高 | 时效性好 | 调度复杂 |

### 同步时机

- 日增量: T+1凌晨执行
- 日全量: T+1凌晨执行
- 小时增量: 每小时整点执行

## 数据质量规范

### 质量维度

| 维度 | 定义 | 检查方法 |
|-----|-----|---------|
| 完整性 | 数据无缺失 | 空值检查 |
| 唯一性 | 数据无重复 | 主键去重 |
| 及时性 | 数据按时到达 | 时间戳检查 |
| 准确性 | 数据准确无误 | 业务规则校验 |
| 一致性 | 数据逻辑一致 | 跨表关联检查 |

### 质量监控

**必检指标:**
- 主键空值数: 0
- 主键重复数: 0
- 日数据量波动: ±30%告警

**监控告警:**
- 数据延迟告警
- 数据量异常告警
- 数据质量告警

## 存储规范

### 文件格式

推荐使用列式存储:
- ORC: Hive原生支持，压缩比高
- Parquet: 跨平台兼容性好

### 压缩格式

| 格式 | 特点 | 适用场景 |
|-----|-----|---------|
| SNAPPY | 压缩快，压缩比中等 | 实时查询场景 |
| ZLIB | 压缩慢，压缩比高 | 归档存储场景 |

### 存储配置

```sql
-- ORC格式配置
STORED AS ORC
TBLPROPERTIES (
  'orc.compress'='SNAPPY',
  'orc.create.index'='true'
)

-- Parquet格式配置
STORED AS PARQUET
TBLPROPERTIES (
  'parquet.compression'='SNAPPY'
)
```

## 最佳实践

### 表设计

1. 保留源系统字段，不做转换
2. 增加ETL加工字段(如etl_time)
3. 合理设置分区粒度
4. 选择合适的存储格式

### 数据同步

1. 优先选择增量同步
2. 全量同步设置保留周期
3. 异常数据记录日志
4. 同步失败支持重试

### 质量保障

1. 建立质量检核规则
2. 设置告警阈值
3. 问题数据隔离处理
4. 定期质量报告
